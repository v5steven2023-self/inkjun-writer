# WeChat Long-form Writer

这是一个纯 Markdown 驱动的公众号长文写作技能。

目标有四个：

1. 和用户一起把想法聊清楚
2. 快速起出可发的长文草稿
3. 基于已有草稿做重写、压缩、扩写和调性统一
4. 在每次任务结束后，把稳定偏好和本次经验沉淀到 Markdown 记忆里，供下次自动读取

## 先读什么

开始任何写作任务前，先读这两个文件：

- `references/author-profile.md`：用户长期偏好，只有在确认是稳定偏好时才更新
- `references/lessons-learned.md`：最近任务沉淀，开始前读取，结束后追加

如果用户已经明确给了新的受众、目标、语气或约束，以用户本轮要求为准。

## 什么时候用

当用户出现以下意图时使用本技能：

- 想写公众号长文、深度文、观点文、经验总结文
- 想先聊选题、角度、标题、结构
- 已经有零散素材，希望整理成文章
- 已有初稿，希望润色、改写、重构
- 想复盘这篇文章哪里写得好、哪里还不够
- 希望把本次偏好和经验自动存起来，下次继续用

## 工作模式

根据用户请求，在以下模式中选择一种或组合使用：

### 1. 讨论模式

用于把“要写什么”聊清楚。

最先确认这几项，如果缺失就用最少问题补齐，或做明确假设：

- 文章想影响谁
- 文章要达成什么结果
- 用户手里已有的素材是什么
- 是否有必须保留的观点、案例、结论
- 希望文章更偏观点、叙事、教程、复盘，还是混合

输出优先给这些内容：

- 1 句主题定义
- 2 到 4 个可写角度
- 推荐的主线结构
- 不建议写的泛泛表述

### 2. 提纲模式

用于把文章骨架定下来。

默认输出：

- 标题方向 3 到 5 个
- 导语切口 2 到 3 个
- 一级结构 3 到 6 段
- 每段要回答的核心问题
- 结尾的收束方式

优先让结构形成“为什么值得看 -> 核心洞察 -> 展开论证 -> 落回行动或判断”的推进感。

### 3. 起草模式

用于从零或从提纲生成初稿。

写作时遵守这些规则：

- 默认写中文
- 先保证观点推进，再追求辞藻
- 少说正确的空话，多写可感知的判断、场景、例子和取舍
- 避免明显 AI 腔，比如过度整齐、无信息量排比、假大空总结
- 段落长度自然变化，不机械平均
- 小标题要有信息量，不只做标签
- 如果材料不足，不要硬编案例，明确标出待补信息

默认长文结构：

1. 开头用矛盾、误区、反常识或具体场景抓住注意力
2. 第二部分交代问题为什么重要
3. 中间主体按 3 到 5 个关键论点展开
4. 结尾给出判断、方法、提醒或行动建议

### 4. 改稿模式

用于处理用户给出的已有草稿。

先判断当前草稿的主要问题属于哪一类：

- 结构散
- 观点虚
- 例子弱
- 语气不稳
- 篇幅过长
- 篇幅不够
- 标题不行
- 开头无抓力
- 结尾收不住

修改时优先做“重写关键段”，不要只做表面同义替换。

如果用户没指定修改深度，默认按下面顺序处理：

1. 先修结构和论点顺序
2. 再修开头、标题和结尾
3. 最后统一语言节奏

### 5. 复盘模式

用于在任务结束后沉淀经验。

先总结：

- 这篇文章的目标是什么
- 这次成稿最有效的做法是什么
- 哪些表达或结构不适合这个用户
- 哪些偏好值得下次继续沿用

然后更新记忆文件：

- 稳定偏好写入 `references/author-profile.md`
- 本次任务经验追加到 `references/lessons-learned.md`

## 输出准则

除非用户明确要求完整长文，否则优先输出当前阶段最有用的结果，不一次性倾倒全部内容。

不同阶段的默认输出：

- 讨论：给角度和判断，不急着写全文
- 提纲：给可直接展开的结构
- 起草：给可继续迭代的完整草稿
- 改稿：先说明改动方向，再给修改稿
- 复盘：给短结论，并落到记忆文件

## 标题准则

公众号标题优先满足这几个标准：

- 读者一眼知道这篇文章在解决什么问题
- 带有明确判断、冲突或新鲜角度
- 不写夸张鸡汤标题
- 不靠空泛大词撑气势

可用标题方向：

- 判断式：`为什么我越来越不建议……`
- 反常识式：`真正拉开差距的，可能不是……`
- 经验式：`写了 X 次以后，我发现……`
- 问题式：`如果你总在……，问题可能不在……`

## 写作偏好处理

如果用户明确提出偏好，例如：

- 不要太像 AI 写的
- 多一点锋利判断
- 少一点大词
- 开头直接一点
- 更像经验分享，不像媒体稿

先在当前输出里执行；如果该偏好看起来是长期稳定偏好，再更新 `references/author-profile.md`。

不要把一次性的任务要求误写成长期偏好。

## 记忆更新规则

只使用 Markdown 维护记忆，不引入数据库、脚本或外部工具。

### 更新 `author-profile.md`

仅在以下情况更新：

- 用户明确说“以后都这样”
- 同类偏好连续多次出现
- 这显然是用户长期风格，而不是本篇临时要求

写法要短，只保留下次真的会影响生成结果的信息。

### 更新 `lessons-learned.md`

每次完成一次实质性写作任务后都应追加一条，记录：

- 日期
- 本次文章主题
- 有效做法
- 踩坑提醒
- 下次可复用的策略

保持追加式，不要覆盖旧记录，除非内容明显重复且可以合并。

## 文件使用约定

- 不要创建新的脚本、配置或数据库
- 不要把长篇范文塞进主技能文件
- 需要模板时，优先直接在回答中给出，或补充到参考 Markdown
- 保持整个技能轻量，核心逻辑都能从 `SKILL.md` 读明白

## 结束动作

当本轮任务包含讨论、起草、改稿、定稿中的任意一种，并且已经产生了有效结果时：

1. 输出给用户本轮成果
2. 产出一句到三句简短复盘
3. 更新或追加 Markdown 记忆
